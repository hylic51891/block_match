# 总体方案（人类开发者版）

## 1.1 项目背景

《回路连连看》已经完成本地可试玩 MVP，核心玩法闭环已成立。当前项目目标不再是验证浏览器原型本身，而是进入 **微信小游戏正式化开发阶段**。

当前项目基础能力已经具备：

- 本地浏览器可运行
- 核心玩法完整
- 路径判定稳定
- 污染机制可用
- 灵能图块机制已落地
- 重排、死局检测、胜负判定已实现
- 存档、日志、埋点、测试框架已初步具备

因此，下一阶段的人类开发者职责，已经从“帮助 AI 做 MVP”转为：

1. 帮助 AI 将项目升级为 **可在微信小游戏环境运行的完整产品**
2. 决定产品形态、商业化节奏和平台接入优先级
3. 审阅跨平台架构改造方案
4. 为 AI 提供平台约束和验收标准
5. 控制微信小游戏版本的范围，避免过早膨胀为大而全项目

---

## 1.2 当前阶段目标

当前阶段目标定义为：

> 将《回路连连看》从“本地可试玩小游戏原型”升级为“可在微信小游戏正式运行的完整项目”。

本阶段重点目标：

- 支持微信小游戏运行环境
- 调整产品结构为 **教学关 + 每日挑战主关**
- 接入微信小游戏基础能力的架构预留与必要实现
- 梳理完整 PRD、技术改造路线与第三期开发计划
- 保留当前 Web 试玩能力作为调试与回归环境

### 本阶段不强制一次性全部完成
- 广告完全接入上线
- 好友榜和地区榜全量上线
- 后端复杂活动系统
- 大量内容型关卡扩充

本阶段更重要的是：

- 产品形态正确
- 微信小游戏技术路线正确
- 架构具备持续迭代能力

---

## 1.3 人类开发者的核心职责

### 1.3.1 搭建双环境开发体系
你需要帮助 AI 维护两套可共存环境：

- **Web 本地调试环境**：用于快速开发、测试、回归
- **微信小游戏运行环境**：用于平台验证、真机预览、提审准备

建议保持：

- 一套核心逻辑
- 两套平台适配层
- 尽量减少平台分叉代码

### 1.3.2 决定产品主结构
你需要明确向 AI 下达以下产品约束：

- 不再以 20 关线性递进作为主流程
- 主流程改为：**教学关 + 每日挑战**
- 首页主入口是今日挑战，不是关卡列表
- 旧 20 关保留为测试/练习/素材池，不是主体验

### 1.3.3 决定平台接入顺序
微信小游戏开发中，建议的接入顺序为：

1. 运行环境适配
2. 输入与渲染验证
3. 存档与本地数据适配
4. 分享能力预留
5. 广告位与激励点设计
6. 登录与排行榜协议设计
7. 云端每日挑战配置

### 1.3.4 决定商业化与社交节奏
不要让 AI 自行扩散到大而全设计。建议你明确以下原则：

- 先验证挑战制是否成立，再接强商业化
- 先做每日挑战，再做好友榜/地区榜
- 先做激励广告触发点设计，再接真实广告 API
- 分享与结果页要比商城、任务系统更优先

## 1.4 产品定义

### 1.4.1 产品名称
回路连连看

### 1.4.2 当前版本定位
一款面向微信小游戏场景的轻度休闲益智挑战游戏。

### 1.4.3 核心产品结构
本阶段产品结构正式调整为：

- **教学关**：首次进入时触发，帮助用户快速理解玩法
- **每日挑战**：每天 1 局，全体玩家共享同一主挑战盘面

### 1.4.4 核心卖点
1. 上手快：基础规则简单
2. 有策略：污染封路 + 灵能图块带来顺序决策
3. 有挑战：每日只打一局主关卡
4. 适合微信小游戏：碎片化、可重试、可分享、可比较

---

## 1.5 产品主策略

### 1.5.1 从关卡制转向挑战制
原先的 20 关线性递进，不再作为主产品结构。原因：

- 微信小游戏更适合短链路挑战
- 用户更容易接受“今天来打一把”
- 当前玩法更适合局面挑战而非章节式内容递进

### 1.5.2 为什么保留旧 20 关
旧 20 关仍有价值，但定位改变为：

- 调试用
- 自动测试素材池
- 练习模式候选内容
- 每日挑战生成参考池

### 1.5.3 当前推荐的产品路径
建议采用：

```text
首次进入 -> 教学关 -> 今日挑战 -> 结果页 -> 重试 / 分享
后续进入 -> 今日挑战 -> 结果页 -> 重试 / 分享
```

## 1.6 技术路线

### 1.6.1 现阶段技术形态
当前项目技术栈仍以：

- React
- TypeScript
- Vite
- Canvas 2D
- Zustand
- Vitest

为主。

### 1.6.2 下一阶段技术目标
不是简单“继续做 Web”，而是建立：

- **核心玩法引擎层**（可复用）
- **Web 适配层**（保留）
- **微信小游戏适配层**（新增）

### 1.6.3 推荐的迁移原则
1. 核心规则层不要重写
2. 平台差异收敛到 adapter 层
3. 先跑通小游戏环境，再接平台能力
4. 保留 Web 端作为高效率开发和自动化测试环境

---

## 1.7 后端定位

当前阶段后端从“未来预留”升级为“应开始定义正式边界”，但仍不建议先做重后端。

### 后端职责建议
- 每日挑战配置下发
- 用户登录态（后续）
- 排行榜（后续）
- 挑战结果上报
- 埋点批量上报
- 分享和挑战元数据管理

### 当前建议
- 先完成 API 契约设计
- 先让客户端能以本地 mock 方式跑通
- 后端与微信平台接入并行推进，不阻塞小游戏前端迁移

---

## 1.8 人类开发者验收标准

你最终应按以下标准验收第三期方向是否正确：

1. 主流程是否已经从“关卡制”切换成“挑战制”
2. 项目是否具备微信小游戏适配层
3. 是否保留 Web 调试能力
4. 教学关和每日挑战结构是否成立
5. 结果页、挑战记录、重试链路是否成立
6. 是否有清晰的微信小游戏接入计划
7. 是否有第三期 PRD 和开发计划文档
8. docs 总览文档是否已经反映新的项目目标

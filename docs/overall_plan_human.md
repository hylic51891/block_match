# 总体方案（人类开发者版）

## 1.1 项目背景

《回路连连看》是一款面向微信小游戏方向的轻度休闲益智游戏。

项目已历经三个阶段：

- **一期（MVP）**：本地浏览器可运行，核心玩法闭环，20 关线性递进
- **二期（挑战制）**：产品结构从 20 关线性递进改为教学关 + 每日挑战，灵能图块 S/T 差异化，限时机制，存档迁移
- **三期（微信小游戏正式化）**：当前阶段，目标从"本地可试玩原型"升级为"可在微信小游戏正式运行的完整项目"

当前项目基础能力已具备：

- 本地浏览器可运行，核心玩法完整
- 路径判定、污染机制、灵能图块、限时机制已实现
- 重排、死局检测、胜负判定、教学关 + 每日挑战已落地
- 存档、日志、埋点、测试框架已具备
- 分享/广告/排行榜/远程配置接口已抽象

因此，当前阶段人类开发者的职责已从"帮助 AI 做 MVP"转为：

1. 帮助 AI 将项目升级为 **可在微信小游戏环境运行的完整产品**
2. 决定产品形态、商业化节奏和平台接入优先级
3. 审阅跨平台架构改造方案
4. 为 AI 提供平台约束和验收标准
5. 控制微信小游戏版本的范围，避免过早膨胀为大而全项目

---

## 1.2 当前阶段目标

当前阶段目标定义为：

> 将《回路连连看》从"本地可试玩小游戏原型"升级为"可在微信小游戏正式运行的完整项目"。

本阶段重点目标：

- 支持微信小游戏运行环境
- 巩固挑战制产品结构（教学关 + 每日挑战 + 限时）
- 接入微信小游戏基础能力的架构预留与必要实现
- 梳理完整 PRD、技术改造路线与开发计划
- 保留当前 Web 试玩能力作为调试与回归环境

### 本阶段不强制一次性全部完成
- 广告完全接入上线
- 好友榜和地区榜全量上线
- 后端复杂活动系统
- 大量内容型关卡扩充

本阶段更重要的是：

- 产品形态正确
- 微信小游戏技术路线正确
- 架构具备持续迭代能力

## 1.3 人类开发者的核心职责

### 1.3.1 搭建双环境开发体系
你需要帮助 AI 维护两套可共存环境：

- **Web 本地调试环境**：用于快速开发、测试、回归
- **微信小游戏运行环境**：用于平台验证、真机预览、提审准备

建议保持：

- 一套核心逻辑
- 两套平台适配层
- 尽量减少平台分叉代码

### 1.3.2 决定产品主结构
你需要明确向 AI 下达以下产品约束：

- 不再以 20 关线性递进作为主流程
- 主流程改为：**教学关 + 每日挑战**
- 首页主入口是今日挑战，不是关卡列表
- 旧 20 关保留为测试/练习/素材池，不是主体验

### 1.3.3 决定平台接入顺序
微信小游戏开发中，建议的接入顺序为：

1. 运行环境适配
2. 输入与渲染验证
3. 存档与本地数据适配
4. 分享能力预留
5. 广告位与激励点设计
6. 登录与排行榜协议设计
7. 云端每日挑战配置

### 1.3.4 决定商业化与社交节奏
不要让 AI 自行扩散到大而全设计。建议你明确以下原则：

- 先验证挑战制是否成立，再接强商业化
- 先做每日挑战，再做好友榜/地区榜
- 先做激励广告触发点设计，再接真实广告 API
- 分享与结果页要比商城、任务系统更优先

## 1.4 产品定义

### 1.4.1 产品名称
回路连连看

### 1.4.2 当前版本定位
一款面向微信小游戏场景的轻度休闲益智挑战游戏。

### 1.4.3 核心产品结构

- **教学关**：首次进入时触发，帮助用户快速理解玩法（含 S/T 灵能图块教学）
- **每日挑战**：每天 1 局，全体玩家共享同一主挑战盘面（限时 180 秒）

### 1.4.4 核心卖点
1. 上手快：基础规则简单
2. 有策略：污染封路 + 灵能图块带来顺序决策
3. 有挑战：每日只打一局主关卡，限时增加紧迫感
4. 适合微信小游戏：碎片化、可重试、可分享、可比较

## 1.5 核心玩法定义

### 1.5.1 基础规则
- 玩家点击两个相同图块
- 若两者可通过合法路径连接，则消除
- 棋盘全部清空则通关
- 若棋盘进入死局且没有重排次数，则失败
- 每日挑战限时 180 秒，超时则失败

### 1.5.2 路径规则
- 路径只能水平或垂直移动
- 默认最多允许 2 次转折
- 路径不能穿过：障碍物、其他 active 图块、污染格
- 灵能图块 S 可穿过污染格且允许 3 折
- 不支持棋盘外圈环绕

### 1.5.3 污染机制
- 每次成功连线后（S 图块匹配除外），路径经过的空白格变为污染格
- 污染格在若干回合内不可被穿过
- 污染格随回合衰减并恢复为空白格
- 每日挑战污染持续 3~4 回合

### 1.5.4 灵能图块
| 类型 | 能力 |
|------|------|
| S（相位灵能） | 可穿过污染格 + 路径允许 3 折 + 匹配时路径不产生污染 |
| T（净化灵能） | 匹配后清除路径周围（曼哈顿距离 1 格）的污染格 |

### 1.5.5 失败条件
- 死局 + 无重排 → 无路可走 / 污染封路 / 重排用尽
- 限时模式超时 → 时间耗尽

### 1.5.6 通关条件
- 所有图块被消除

## 1.6 产品主策略

### 1.6.1 从关卡制转向挑战制
原先的 20 关线性递进，不再作为主产品结构。原因：

- 微信小游戏更适合短链路挑战
- 用户更容易接受"今天来打一把"
- 当前玩法更适合局面挑战而非章节式内容递进

### 1.6.2 为什么保留旧 20 关
旧 20 关仍有价值，但定位改变为：

- 调试用
- 自动测试素材池
- 练习模式候选内容
- 每日挑战生成参考池

### 1.6.3 当前推荐的产品路径
```text
首次进入 -> 教学关 -> 今日挑战 -> 结果页 -> 重试 / 分享
后续进入 -> 今日挑战 -> 结果页 -> 重试 / 分享
```

## 1.7 技术路线

### 1.7.1 当前技术栈
- React + TypeScript + Vite + Canvas 2D + Zustand + Vitest

### 1.7.2 下一阶段技术目标
建立三层架构：

- **核心玩法引擎层**（可复用，不依赖平台）
- **Web 适配层**（保留）
- **微信小游戏适配层**（新增）

### 1.7.3 推荐的迁移原则
1. 核心规则层不要重写
2. 平台差异收敛到 adapter 层
3. 先跑通小游戏环境，再接平台能力
4. 保留 Web 端作为高效率开发和自动化测试环境

### 1.7.4 当前目录结构
```text
src/
  app/                # 入口、路由
  scenes/             # Home / Battle / Result
  core/
    board/            # 棋盘模型 + 生成器
    level/            # 关卡加载 + 20关数据
    challenge/        # 教学关/每日挑战 Provider
    rules/            # PathFinder + MatchResolver
    systems/          # Deadlock / Shuffle / Pollution / Hint / WinLose
    engine/           # GameEngine（状态机驱动）
  render/             # Canvas 渲染器
  store/              # game-store + challenge-store + ui-store
  infra/
    storage/          # localStorage 存档
    share/            # IShareService + MockShare
    ad/               # IAdService + MockAd
    leaderboard/      # ILeaderboardService + Mock
    remote-config/    # IRemoteChallengeConfig + Local
    logger / telemetry / error-reporter
  hooks/              # Canvas / Click / GameLoop
  components/         # BattleHUD / LevelCard / DebugPanel
  types/              # 全局类型
  assets/             # 颜色配置
  tests/              # 83 个测试用例
```

### 1.7.5 三期目标目录结构
```text
src/
  app/                # 启动、路由、模式切换
  scenes/             # Home / Tutorial / DailyChallenge / Result
  core/               # 纯玩法逻辑（不变）
  platform/           # Web / WeChat Mini Game 适配层
  services/           # challenge、storage、share、ad、auth 抽象服务
  store/              # runtime 与 UI 状态
  infra/              # logger、telemetry、error-reporter
  tests/              # 核心逻辑与集成测试
```

## 1.8 后端定位

当前阶段后端从"未来预留"升级为"应开始定义正式边界"，但仍不建议先做重后端。

### 后端职责建议
- 每日挑战配置下发
- 用户登录态（后续）
- 排行榜（后续）
- 挑战结果上报
- 埋点批量上报
- 分享和挑战元数据管理

### 当前建议
- 先完成 API 契约设计
- 先让客户端能以本地 mock 方式跑通
- 后端与微信平台接入并行推进，不阻塞小游戏前端迁移

## 1.9 人类开发者验收标准

你最终应按以下标准验收项目方向是否正确：

1. 主流程是否已经从"关卡制"切换成"挑战制"
2. 项目是否具备微信小游戏适配层
3. 是否保留 Web 调试能力
4. 教学关和每日挑战结构是否成立
5. 结果页、挑战记录、重试链路是否成立
6. 是否有清晰的微信小游戏接入计划
7. 限时机制和灵能图块差异化是否生效
8. docs 总览文档是否已经反映新的项目目标
